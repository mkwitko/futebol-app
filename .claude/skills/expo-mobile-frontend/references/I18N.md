# I18N.md — Internacionalização (React Native)

> O app começa **apenas em pt-BR**, mas toda a estrutura está pronta para adicionar idiomas sem refatoração. Tudo que vai pra tela passa por `t("chave")` — nada de string hardcoded. Idioma inicial vem do **device** via `expo-localization` (`getLocales()`), não do browser. Contexto geral em [`../SKILL.md`](../SKILL.md).

---

## 1. Stack

- **i18next** — engine de tradução, pluralização, interpolação.
- **react-i18next** — bindings React (`useTranslation`, `<Trans>`) — funciona igual em RN.
- **expo-localization** — locale/região/timezone do **device** (`getLocales()`, `getCalendars()`). Substitui `navigator.language` e o `i18next-browser-languagedetector` do web.
- **Recursos bundlados** — os JSONs de tradução são **importados** e embutidos no bundle (via `resources`). Em RN não há `public/` servido por HTTP, então **não usamos** `i18next-http-backend` nem lazy-load por rede.

```bash
pnpm add i18next react-i18next
npx expo install expo-localization
```

> O namespace `zod.json` existe e é carregado, mas **`zod-i18n-map` ainda não está instalado/wired** (`initI18n` não chama `z.setErrorMap`). Quando os forms passarem a usar Zod (ver [`FORMS.md`](./FORMS.md)), adicione a lib e ligue o map.

## 2. Configuração

Sem language detector de browser: lemos o idioma do device com `getLocales()` e caímos no default se não suportado. Recursos vêm de um barrel de imports (bundlados).

```ts
// src/lib/i18n/index.ts
import i18n from "i18next";
import { initReactI18next } from "react-i18next";
import { getLocales } from "expo-localization";
import { resources } from "./resources"; // barrel com os JSONs importados

export const DEFAULT_LANGUAGE = "pt-BR";
export const SUPPORTED_LOCALES = ["pt-BR"] as const; // adicione "en"/"es" no futuro
export const NAMESPACES = [
  "common",
  "zod",
  "users",
  "bookings",
] as const;

/** Idioma inicial a partir do device, com fallback no default. */
function resolveInitialLanguage(): string {
  // getLocales() é síncrono e garante ≥ 1 elemento (o preferido primeiro).
  const deviceTag = getLocales()[0]?.languageTag ?? DEFAULT_LANGUAGE; // ex.: "pt-BR"
  const supported = SUPPORTED_LOCALES as readonly string[];
  if (supported.includes(deviceTag)) return deviceTag;
  // tenta casar só pelo idioma ("pt" de "pt-PT" → "pt-BR")
  const base = deviceTag.split("-")[0];
  return supported.find((l) => l.startsWith(base)) ?? DEFAULT_LANGUAGE;
}

export async function initI18n() {
  if (i18n.isInitialized) return i18n; // idempotente

  await i18n
    .use(initReactI18next)
    .init({
      resources,                             // JSONs bundlados
      lng: resolveInitialLanguage(),         // do device (expo-localization)
      fallbackLng: DEFAULT_LANGUAGE,
      supportedLngs: SUPPORTED_LOCALES as unknown as string[],
      ns: NAMESPACES,
      defaultNS: "common",
      interpolation: { escapeValue: false }, // React já escapa
      returnNull: false,
      compatibilityJSON: "v4",               // regras de plural CLDR
    });

  return i18n;
}

export { default as i18n } from "i18next";
```

> **Android:** `getLocales()` pode mudar se o usuário troca o idioma do sistema e volta pro app. Se precisar reagir a isso, re-resolva o idioma num listener de `AppState` (foreground) e chame `i18n.changeLanguage(...)`. No boot inicial o exemplo acima basta.

O barrel de recursos importa os JSONs (bundlados pelo Metro):

```ts
// src/lib/i18n/resources.ts
import commonPt from "./locales/pt-BR/common.json";
import zodPt from "./locales/pt-BR/zod.json";
import usersPt from "./locales/pt-BR/users.json";
import bookingsPt from "./locales/pt-BR/bookings.json";

export const resources = {
  "pt-BR": {
    common: commonPt,
    zod: zodPt,
    users: usersPt,
    bookings: bookingsPt,
  },
} as const;
```

## 3. Estrutura de arquivos

Em RN os JSONs ficam **dentro de `src/`** (para o Metro bundlar via `import`), não em `public/`:

```
src/
└── lib/
    └── i18n/
        ├── index.ts             # init (initI18n, NAMESPACES, DEFAULT_LANGUAGE, SUPPORTED_LOCALES)
        ├── resources.ts         # barrel: importa e agrupa os JSONs
        ├── types.d.ts           # type augmentation (ver §6)
        └── locales/
            └── pt-BR/           # único idioma hoje
                ├── common.json  # botões, ações comuns, mensagens globais
                ├── zod.json     # mensagens de erro do Zod (lib ainda não wired)
                ├── users.json
                └── bookings.json
```

**Decisão arquitetural:**

Cada módulo tem seu namespace em `src/lib/i18n/locales/pt-BR/<modulo>.json`. Ao criar um namespace novo: (1) adicione o JSON, (2) importe-o em `resources.ts`, (3) adicione o nome no array `NAMESPACES` em `index.ts`. Três passos, nenhum toque em componente.

## 4. Bootstrap do app (Expo Router)

Não há `main.tsx`/`createRoot` — o entry é o **root layout** do Expo Router (`src/app/_layout.tsx`). Inicializamos o i18n antes de renderizar as rotas, segurando a splash screen até estar pronto.

```tsx
// src/app/_layout.tsx (trecho — ver AUTH.md para o bootstrap completo de providers)
import { useEffect, useState } from "react";
import { Slot } from "expo-router";
import * as SplashScreen from "expo-splash-screen";
import { I18nextProvider } from "react-i18next";
import { i18n, initI18n } from "@/lib/i18n";
import "@/lib/i18n/types"; // ativa o type augmentation

SplashScreen.preventAutoHideAsync();

export default function RootLayout() {
  const [ready, setReady] = useState(false);

  useEffect(() => {
    initI18n().then(() => setReady(true)).finally(() => SplashScreen.hideAsync());
  }, []);

  if (!ready) return null;

  return (
    <I18nextProvider i18n={i18n}>
      {/* QueryClientProvider > AuthProvider > <Stack/> */}
      <Slot />
    </I18nextProvider>
  );
}
```

`initI18n()` é idempotente (guard `i18n.isInitialized`).

### 5.2 Múltiplos namespaces

```tsx
const { t } = useTranslation(["users", "common"]);

t("list.title");              // users.list.title (default = primeiro NS)
t("common:actions.cancel");   // common.actions.cancel (prefixo explícito)
```

### 5.3 Interpolação

```json
{ "welcome": "Olá, {{name}}!" }
```

```tsx
t("welcome", { name: user.name });
```

### 5.4 Pluralização (CLDR)

```json
{
  "items_one": "{{count}} item",
  "items_other": "{{count}} itens"
}
```

```tsx
t("items", { count: 1 });    // "1 item"
t("items", { count: 5 });    // "5 itens"
```

### 5.5 `<Trans>` para texto com markup

Quando a tradução tem trechos formatados (bold, link para outra tela):

```tsx
import { Trans } from "react-i18next";
import { Text } from "react-native";

<Trans i18nKey="terms.accept" ns="auth">
  Aceito os <Text className="text-primary underline">termos de uso</Text>.
</Trans>
```

```json
{ "terms": { "accept": "Aceito os <1>termos de uso</1>." } }
```

Os números (`<1>`) correspondem à posição do filho. Em RN os "elementos" são componentes RN (`Text`/`Pressable`), não tags HTML.

## 6. Type-safety

i18next suporta typing via module augmentation. Garante que `t("chave")` é validado em compile time.

```ts
// src/lib/i18n/types.d.ts
import "i18next";
import type common from "./locales/pt-BR/common.json";
import type users from "./locales/pt-BR/users.json";
import type bookings from "./locales/pt-BR/bookings.json";
import type zodNs from "./locales/pt-BR/zod.json";

declare module "i18next" {
  interface CustomTypeOptions {
    defaultNS: "common";
    resources: {
      common: typeof common;
      users: typeof users;
      bookings: typeof bookings;
      zod: typeof zodNs;
      // ... um por namespace de NAMESPACES
    };
  }
}
```

Agora `t("nonexistent.key")` é erro de tipo. Editor autocompleta chaves. O arquivo deve listar **todos** os namespaces de `NAMESPACES`.

```json
{
  "errors": {
    "invalid_type": "Tipo inválido: esperado {{expected}}, recebido {{received}}",
    "invalid_type_received_undefined": "Campo obrigatório",
    "invalid_string": {
      "email": "E-mail inválido",
      "url": "URL inválida",
      "uuid": "UUID inválido"
    },
    "too_small": {
      "string": {
        "inclusive": "Mínimo {{minimum}} caracteres",
        "not_inclusive": "Mais de {{minimum}} caracteres"
      },
      "number": {
        "inclusive": "Mínimo {{minimum}}",
        "not_inclusive": "Maior que {{minimum}}"
      }
    },
    "too_big": {
      "string": {
        "inclusive": "Máximo {{maximum}} caracteres"
      }
    },
    "invalid_enum_value": "Valor inválido"
  }
}
```

Lib `zod-i18n-map` lê isso automaticamente (quando wired).

### 7.2 Mensagens customizadas no schema

Prefira **chaves i18n como string** na mensagem do Zod — o componente `<FieldError>` traduz no render (ver [`FORMS.md`](./FORMS.md) §2.2). Isso evita o problema de `i18n.t()` ser chamado antes do i18n estar pronto:

```ts
// src/schemas/users/create-user.schema.ts
import { z } from "zod";

export const createUserSchema = z.object({
  // A "message" é uma chave; o <FieldError> faz t(message).
  name: z.string().min(1, { message: "users:form.errors.name.required" }),
  email: z.string().email(), // mensagem padrão via zod-i18n-map → "E-mail inválido"
});
```

Assim o schema é um módulo puro (sem side-effect de tradução no import) e a tradução acontece só na hora de exibir.

## 8. Formatação de datas, números, moeda

**Datas têm doc próprio:** [`DATES.md`](./DATES.md) cobre formatação com timezone (do device via `expo-localization`), conversão ↔ UTC, input do usuário e armadilhas. Use sempre `useFormat()` (em `src/hooks/common/use-format.ts`) — ele já injeta o timezone do device.

**Números e moeda** usam `Intl` nativo. O runtime do Expo (Hermes) inclui ICU completo, então `Intl.NumberFormat` funciona em RN — zero KB adicionais.

```ts
// src/lib/format-numbers.ts
export function formatCurrency(value: number, locale = "pt-BR", currency = "BRL"): string {
  return new Intl.NumberFormat(locale, { style: "currency", currency }).format(value);
}

export function formatNumber(value: number, locale = "pt-BR", options?: Intl.NumberFormatOptions): string {
  return new Intl.NumberFormat(locale, options).format(value);
}

export function formatPercent(value: number, locale = "pt-BR"): string {
  return new Intl.NumberFormat(locale, { style: "percent", maximumFractionDigits: 1 }).format(value);
}
```

Hook reativo ao idioma ativo (delega datas pro `src/lib/datetime/format.ts`):

```ts
// src/hooks/common/use-format.ts
import { useTranslation } from "react-i18next";
import { useTimezone } from "./use-timezone";
import * as dateFmt from "@/lib/datetime/format";
import * as numFmt from "@/lib/format-numbers";

export function useFormat() {
  const { i18n } = useTranslation();
  const tz = useTimezone();       // timezone do device (expo-localization) — ver DATES.md
  const locale = i18n.language;

  return {
    // Datas — delegam pra src/lib/datetime/format.ts (recebem o tz do device)
    date: (input: string | Date) => dateFmt.formatDate(input, tz),
    dateTime: (input: string | Date) => dateFmt.formatDateTime(input, tz),
    time: (input: string | Date) => dateFmt.formatTime(input, tz),
    smart: (input: string | Date, now?: Date) => dateFmt.formatSmart(input, tz, now),
    relative: (input: string | Date, now?: Date) => dateFmt.formatRelative(input, locale, now),
    range: (start: string | Date, end: string | Date) => dateFmt.formatRange(start, end, tz),

    // Números — Intl nativo (Hermes ICU)
    currency: (value: number, currency?: string) => numFmt.formatCurrency(value, locale, currency),
    number: (value: number, options?: Intl.NumberFormatOptions) => numFmt.formatNumber(value, locale, options),
    percent: (value: number) => numFmt.formatPercent(value, locale),
  };
}
```

## 9. Switch de idioma

```tsx
// src/components/settings/language-switcher.tsx
import { View, Text, Pressable } from "react-native";
import { useTranslation } from "react-i18next";
import { SUPPORTED_LOCALES } from "@/lib/i18n";
import { cn } from "@/lib/utils";

const LABELS: Record<(typeof SUPPORTED_LOCALES)[number], string> = {
  "pt-BR": "Português (Brasil)",
  // "en": "English",
  // "es": "Español",
};

export function LanguageSwitcher() {
  const { i18n } = useTranslation();
  if (SUPPORTED_LOCALES.length <= 1) return null; // não mostra se só 1 idioma

  return (
    <View className="gap-2">
      {SUPPORTED_LOCALES.map((lng) => (
        <Pressable
          key={lng}
          accessibilityRole="button"
          onPress={() => i18n.changeLanguage(lng)}
          className={cn(
            "rounded-lg border border-input px-4 py-3",
            i18n.language === lng && "border-primary bg-primary/10",
          )}
        >
          <Text className="text-foreground">{LABELS[lng]}</Text>
        </Pressable>
      ))}
    </View>
  );
}
```

Adicionar idioma novo no futuro:
1. Crie `src/lib/i18n/locales/<lng>/*.json`.
2. Importe em `resources.ts` e adicione o locale a `SUPPORTED_LOCALES`.
3. Adicione label em `LABELS`.

Pronto. Sem mudar mais nada. (Opcional: persistir a escolha do usuário em `AsyncStorage` e ler no `resolveInitialLanguage`.)

## 10. Plurais de pt-BR vs en — sutilezas

i18next usa as regras CLDR (`compatibilityJSON: "v4"`). pt-BR e en têm regras **diferentes**:

- en: `_one` (1) + `_other` (resto).
- pt-BR: `_one` (0-1) + `_other` (resto). Sim, zero é singular em português ("0 item" parece estranho — mas o padrão CLDR é assim).

Para "0 itens" em português, use chave própria:

```json
{
  "items_zero": "Nenhum item",
  "items_one": "{{count}} item",
  "items_other": "{{count}} itens"
}
```

(i18next reconhece `_zero` mesmo onde CLDR não define, se você fornece.)

## 12. Testes com i18n

> **No repo, `test/setup.ts` inicializa o i18n real** (via `initI18n`), então os testes carregam as traduções bundladas de verdade. O mock a seguir é uma alternativa quando se quer asserts por chave; ver [`TESTING.md`](./TESTING.md).

Mock simples no setup do jest — devolve a chave em vez do texto. Testes ficam previsíveis sem depender do conteúdo das traduções.

```ts
// test/mocks/i18n.ts
jest.mock("react-i18next", () => ({
  useTranslation: () => ({
    t: (key: string) => key,         // retorna a chave
    i18n: { language: "pt-BR", changeLanguage: jest.fn() },
  }),
  Trans: ({ i18nKey }: { i18nKey: string }) => i18nKey,
  initReactI18next: { type: "3rdParty", init: () => {} },
}));
```

Em `test/setup.ts`:

```ts
import "./mocks/i18n";
```

Asserts ficam:

```tsx
expect(screen.getByText("users:list.title")).toBeTruthy();
```

Não é o mais "realista", mas é estável e rápido. Para testes onde a tradução **importa** (validação de erro Zod traduzido), inicialize i18n real com fixture mínima.

## 13. CLAUDE Code — regras

- **Nada hardcoded.** Todo texto visível é `t("chave")` ou `<Trans>` (nunca literal num `<Text>`).
- **Mensagens de erro de API** vêm do backend e **não passam por i18n no front** — backend já manda em pt-BR (ou no idioma do `Accept-Language` se implementarmos).
- **Logs / erros console** ficam em inglês (são pra dev, não usuário).
- **Comentários no código** em inglês (padrão).
- **Strings dentro de schemas Zod** sempre via chave i18n, traduzidas no `<FieldError>` (ver §7.2 e [`FORMS.md`](./FORMS.md)).
- **Datas, moedas, números** sempre via `useFormat()`. Datas em particular têm regras complexas de timezone — leia [`DATES.md`](./DATES.md) antes de tocar.
- **Chave de tradução** segue o padrão `<feature>:<contexto>.<sub>`. Ex: `users:form.fields.email`, `common:actions.save`.

## 14. Adicionar inglês depois — checklist

1. Crie `src/lib/i18n/locales/en/` espelhando estrutura de pt-BR.
2. Traduza cada JSON.
3. Importe os JSONs em `resources.ts` e adicione `"en"` a `SUPPORTED_LOCALES`.
4. Adicione `"en": "English"` em `LABELS` no `LanguageSwitcher`.
5. `resolveInitialLanguage()` já passa a respeitar device em inglês automaticamente.

**Sem mudanças no código de componentes.**

## 15. Checklist i18n

- [ ] Nenhum texto hardcoded em componentes (`<Text>` sempre com `t()`)
- [ ] Toda chave existe no JSON do namespace certo
- [ ] Namespace novo: JSON criado + importado em `resources.ts` + nome em `NAMESPACES`
- [ ] Pluralização usa `_one`/`_other` (+ `_zero` se relevante em pt-BR)
- [ ] Datas/moedas via `useFormat()`
- [ ] Mensagens Zod via `zod-i18n-map` ou chave própria traduzida no `<FieldError>`
- [ ] Idioma inicial resolvido de `getLocales()` (device), com fallback no default
- [ ] Adicionar idioma novo não exige refatorar componente
